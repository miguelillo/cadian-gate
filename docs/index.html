01

LOGIN METHODS

● CORE

PATTERN UNLOCK

pattern + password → keyed HMAC identity

The user draws a pattern on a 6×6 grid (4+ distinct dots) and types a password. The pair is hashed into a keyed HMAC-SHA256 identity — login is a single O(1) lookup and the raw credentials are never stored.

● NEW

USER LOGIN

traditional username + password

Classic account sign-in: a login name (3–32 chars, [a-z0-9._-], case-insensitive) verified against a bcrypt hash. Ideal for guests or anyone who prefers typing over drawing.

The tab appears automatically on the login page once at least one password user exists.

● OVERRIDE

BREAK-GLASS

single-operator env credentials

Emergency sign-in backed by environment variables — no database required. Auto-disables once a store user exists (unless forced on) and stays available at zero users so the first account can be bootstrapped.

● 2FA

TOTP

authenticator second factor

Optional per-user 6-digit codes (SHA1 / 30s, Google-Authenticator compatible). Enrollment is proven at user creation: the secret only activates after a live code verifies.

● RECOVERY

BACKUP CODES

single-use bcrypt-hashed recovery

Pre-generated recovery codes accepted in place of a TOTP code. Each one is stored as a bcrypt hash and permanently consumed on first use.

● SHIELD

IP LOCKOUT + AUDIT

10 failures → 15 min lock (tunable)

Every attempt — success, invalid or locked — is recorded to the access_events collection with an optional client fingerprint (IP, ISP, device, display, timezone…).

BACKEND ▸ ASP.NET CORE

// INSTALL + WIRE UP

PROGRAM.CSC# ▪ net10.0
using PatternAuth;

builder.Services.AddPatternAuth(builder.Configuration, o =>
{
    // The only hook most hosts need — where Mongo comes from:
    o.MongoDatabaseFactory = sp => sp.GetRequiredService<MyMongoHolder>().Database;
    // o.CookieName = "my_token"; o.BasePath = "/api/auth"; …
});

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();
app.MapPatternAuthEndpoints();       // /api/auth: me, config, setup-totp, login, logout
app.MapPatternAuthUsersEndpoints();  // /api/users: list, create, delete, new-totp-secret
⚠ RATE LIMITING — POST /login carries RequireRateLimiting("login") by default. Define that policy in your host (or set LoginRateLimitPolicy = null).
⚠ MONGO OPTIONAL — without a database the store is empty: pattern and username login never match and break-glass stays available. Handy for CI and first boot.

CONFIGURATION

KEYREQPURPOSE
JWT_SECRETREQUIRED Signs the session JWT; also the fallback HMAC key
HOMELAB_USER_ID_KEYrecommended HMAC key for pattern-user identities (keep distinct from the JWT secret)
HOMELAB_PASSWORD_HASHoptional bcrypt hash for the break-glass password sign-in
HOMELAB_TOTP_ENABLED / HOMELAB_TOTP_SECREToptional TOTP for the break-glass account
HOMELAB_BACKUP_CODESoptional base64 of comma-joined bcrypt hashes (single-use recovery codes)
HOMELAB_BREAKGLASS_ENABLEDoptional Force break-glass on. Otherwise it auto-disables once ≥1 store user exists

Key names are overridable via PatternAuthOptions, along with cookie name, base paths, token lifetime, grid size, minimum pattern length, lockout thresholds and Mongo collection names.

07

SECURITY MODEL

// PATTERN IDENTITY

IDENTITY DERIVATIONHMAC-SHA256
_id = HMAC-SHA256( key = HOMELAB_USER_ID_KEY,
                  msg = normalize(pattern) "␟" password )

Deterministic — login is an O(1) lookup — yet not brute-forceable offline without the server key, even if the database leaks. The raw pattern and password never touch disk.

// PASSWORD IDENTITY

VERIFICATIONBCRYPT wf=12
user = db.users.findOne({ username: normalize(name) })
ok   = bcrypt.verify(password, user.passwordHash)

Usernames are normalized (lowercase, 3–32 chars of [a-z0-9._-]) and unique. Password users get a random identifier — nothing about the credential is derivable from the id.
